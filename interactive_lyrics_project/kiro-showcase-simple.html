How LearningSong Was Created Using Spec-Driven Development

üéµ What is LearningSong?

The Problem

Educational content is boring and hard to remember. Students struggle to memorize facts, concepts, and theories. Traditional study methods are inefficient.

The Solution

LearningSong transforms educational content into memorable songs using AI. It merges music and learning‚ÄîSuno API generates professional songs, LangChain converts study material into singable lyrics.

Why It's Interesting

Songs are 10x more memorable than text. Dual variations let users compare AI-generated songs. Karaoke-style sync with word-level highlighting makes learning interactive and engaging.

The Three-Page Flow

  • Page A (Text Input): Paste content, toggle Google Search grounding, see rate limits (3 songs/day)
  • Page B (Lyrics Editing): Review AI-generated lyrics, select music style (8 options: Pop, Rap, Folk, Electronic, Rock, Jazz, Children's, Classical)
  • Page C (Song Playback): Play songs, switch variations, enjoy karaoke-style synchronized lyrics with word-level highlighting

üìã Spec-Driven Development

LearningSong was built entirely through spec-driven development. Each feature was designed before code was written.

The 10 Specs (In Order)

  • 1. project-setup: Foundation‚ÄîReact/FastAPI architecture, Firebase integration, dev environment (5 properties)
  • 2. page-a-text-input: Home page‚Äîtext input, Google Search grounding, rate limit display (8 properties)
  • 3. page-b-lyrics-editing: Lyrics editor‚ÄîAI-generated lyrics, 8 music styles, song generation (6 properties)
  • 4. page-c-song-playback: Playback page‚Äîaudio player, lyrics display, metadata (7 properties)
  • 5. dual-song-selection: Dual variations‚ÄîSongSwitcher, variation storage, preference persistence (28 properties - most complex!)
  • 6. timestamped-lyrics-sync: Karaoke sync‚Äîword-level highlighting, binary search O(log n), auto-scroll (12 properties)
  • 7. song-playback-improvements: Offset adjustment, song history, LRC export (9 properties)
  • 8. page-b-debugging: Debugging pipeline issues and UI refinements (5 properties)
  • 9. e2e-chrome-devtools-testing: End-to-end testing with Chrome DevTools MCP (10 properties)
  • 10. implementation-analysis: Cross-cutting concerns and architecture review (6 properties)

Why Spec-Driven Was Effective

Before specs: Vague requirements ‚Üí unclear implementation ‚Üí constant back-and-forth ‚Üí bugs in production

With specs: Clear requirements ‚Üí designed architecture ‚Üí implementation checklist ‚Üí Kiro executes systematically ‚Üí edge cases caught early

Example: Dual-song-selection had 28 correctness properties covering edge cases like concurrent switches, malformed API responses, and offline scenarios. All designed before code was written. Kiro implemented them correctly on the first try.

üîó Agent Hooks: Automated Workflows

Agent hooks automate repetitive tasks. LearningSong uses 4 hooks to maintain code quality.

1. ESLint Report Generator

Purpose: Automated linting with prioritized error reporting

Content: Runs ESLint, saves raw output to ./temp/, generates formatted checklist in ./report/lint-checking/ with issues grouped by file and prioritized (Urgent ‚Üí Low Risk)

Goal: Transform raw linting output into actionable todo lists

2. Oxlint Report Generator

Purpose: Fast linting for daily development

Content: Runs Oxlint (faster than ESLint), generates daily report with timestamp, useful for quick feedback during development

Goal: Catch issues early without slowing down development

3. Frontend Test Report

Purpose: Automated Jest test execution and reporting

Content: Runs Jest tests, generates timestamped report in ./report/frontend-test/, organizes errors as todo lists, shows test coverage

Goal: Ensure tests pass before committing

4. Backend Test Report

Purpose: Automated pytest execution with coverage

Content: Runs pytest with coverage, generates comprehensive report in ./report/, shows coverage percentage and uncovered lines, highlights failing tests

Goal: Maintain test coverage and catch regressions

Impact: Instead of manually running tests and parsing output, developers click a button and get actionable reports. This transformed the workflow from chaotic to systematic.

üìö Steering Documents

Steering documents keep Kiro aligned with project conventions and architecture.

product.md

Content: Core flow, features, MVP constraints (anonymous auth, 3 songs/day, 48-hour retention, max 10,000 words)

Goal: Ensure Kiro understands the product vision and constraints

structure.md

Content: Project layout, frontend/backend directory structure, naming conventions, path aliases (@/ ‚Üí ./src/)

Goal: Ensure files are created in the right places with consistent naming

tech.md

Content: Technology stack, quick commands, dev workflow, API docs location

Goal: Ensure Kiro uses correct libraries and follows dev workflow

Why Steering Matters: Steering documents are the "project constitution." They prevent Kiro from making inconsistent decisions. Before adding structure.md, Kiro sometimes created files in wrong directories. After adding it, every generated file landed exactly where it should.

üîå MCP Integrations: Extended Capabilities

MCP (Model Context Protocol) servers extend Kiro's capabilities beyond code generation.

Fetch MCP

Purpose: Access external API documentation

Content: Fetches Suno API docs, library docs (TanStack Query, Zustand, React 19), up-to-date API specs

Goal: Enable Kiro to work with external APIs without manual copy-paste

Context7 MCP

Purpose: Library-specific knowledge

Content: Version-specific docs for React 19, FastAPI, shadcn/ui, breaking changes, best practices

Goal: Reduce hallucination on API usage

Chrome DevTools MCP

Purpose: E2E testing and debugging

Content: Visual debugging, network monitoring, console monitoring, page navigation automation

Goal: Enable comprehensive E2E testing without separate Playwright/Puppeteer scripts

Impact: MCPs transformed Kiro from a code generator into a full development partner. Without them, Kiro would need manual API documentation, library docs, and separate testing tools. With them, Kiro has everything needed to build complex features end-to-end.
